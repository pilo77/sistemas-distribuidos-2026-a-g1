# Architecture Decision Records (ADR)
## Proyecto: RematePOS

Este documento recopila las decisiones arquitectónicas del sistema RematePOS.  
Cada ADR describe una decisión técnica importante, su contexto, impacto y aplicación dentro del proyecto.

---

# ADR-001 — Adoptar arquitectura por capas / hexagonal ligera en cada microservicio

**Responsable:** Carlos Andrés Villamil Yusunguaira  
**Tipo:** Patrón estructural  
**Estado:** Propuesto  

## Contexto
El microservicio de productos puede crecer rápidamente. Si se mezcla controlador, lógica de negocio y acceso a datos en la misma clase, el código se vuelve difícil de mantener.

## Decisión
Separar cada microservicio en capas claras:

- controller  
- application/service  
- domain  
- infrastructure/repository  

## Impacto
Beneficios:
- Mejor mantenibilidad
- Código más organizado
- Facilita pruebas

Costos:
- Reorganización de clases existentes

## Aplicación en RematePOS
Mover la lógica del microservicio de productos para que el controller solo gestione solicitudes HTTP.

---

# ADR-002 — Usar API Gateway como punto único de entrada

**Responsable:** Kevin Santiago Cuesta Hernández  
**Tipo:** Patrón estructural  
**Estado:** Propuesto  

## Contexto
Si el sistema evoluciona hacia microservicios, no es conveniente que el frontend consuma cada servicio directamente.

## Decisión
Implementar un API Gateway que centralice el acceso al sistema.

## Impacto
Beneficios:
- Control centralizado
- Seguridad
- Simplificación del consumo de APIs

Costos:
- Agrega un componente adicional

## Aplicación en RematePOS
El frontend consumirá el Gateway en lugar de consumir directamente el microservicio de productos.

---

# ADR-003 — Base de datos por microservicio

**Responsable:** Kevin Santiago Cuesta Hernández  
**Tipo:** Patrón estructural  
**Estado:** Propuesto  

## Contexto
Compartir una sola base de datos entre microservicios genera acoplamiento.

## Decisión
Cada microservicio debe tener su propia base de datos o esquema independiente.

## Impacto
Beneficios:
- Mayor independencia
- Escalabilidad

Costos:
- Consultas entre servicios más complejas

## Aplicación en RematePOS
El microservicio de productos administrará su propia base de datos.

# ADR-005 — Validación de datos en backend con Bean Validation

**Responsable:** Carlos Andrés Villamil Yusunguaira  
**Tipo:** Patrón de diseño  
**Estado:** Propuesto  

## Contexto
Los datos enviados desde el frontend pueden ser inválidos.

## Decisión
Implementar validaciones con Bean Validation.

Ejemplos:
- `@NotBlank`
- `@NotNull`
- `@Positive`
- `@Size`

## Impacto
Beneficios:
- Mayor integridad de datos
- API más robusta

## Aplicación en RematePOS
Validar creación y edición de productos.

# ADR-006 — Manejo global de errores con @RestControllerAdvice

**Responsable:** Carlos Andrés Villamil Yusunguaira  
**Tipo:** Patrón comportamental  
**Estado:** Propuesto  

## Contexto
Si cada controlador maneja errores de forma distinta, la API se vuelve inconsistente.

## Decisión
Centralizar el manejo de errores con `@RestControllerAdvice`.

## Impacto
Beneficios:
- Respuestas uniformes
- Código más limpio

## Aplicación en RematePOS
Estructura estándar de error:

- timestamp  
- status  
- error  
- message  
- path

# ADR-007 — Delimitar transacciones en capa de servicio
